# Design: Diagram / Layout / Prototype refactor

> **Status:** Approved design, not yet implemented.
> **Purpose:** Drive a multi-session refactor. This document is self-contained — a fresh
> session should be able to execute it without prior chat context. Companion to
> `docs/flow-diagram-editor-spec.md` (the original spec); where they disagree, **this document
> wins** for the areas it covers (domain model §5, styling §8, prototypes/creation §9, views §7.2).
> Update the **Progress checklist** at the bottom as phases land.

## 1. Why

The shipped model over-layers styling (palette → prototype → StyleProfile → entity override →
node/edge override) and overloads "views" with palette/filter/positions. We are collapsing it to
the simplest model that still supports reusable styles, per-diagram looks, and multiple position
arrangements:

- A **Graph** owns entities, relationships, **NodePrototypes**, **EdgePrototypes**, palettes.
- An **Entity** links a `nodePrototypeId` (its default node style). A **Relationship** links an
  `edgePrototypeId`. Relationships are first-class and **browsable**.
- A **Diagram** (was *Board*) owns **subgraph membership *and* styling**: a node's full style is on
  the `node` row, an edge's on the `edge` row.
- A **Layout** (was *View*) is **only positioning** — node positions for a diagram, hand-placed or
  generated by **Dagre**. A diagram has ≥1 layout and can keep several side by side.

## 2. Decision log (locked)

| # | Decision |
|---|----------|
| D1 | **Diagram = subgraph** (membership) **+ styling**. Rename `Board`→`Diagram`. |
| D2 | **Layout = positions only**. Rename `View`→`Layout`. A diagram has ≥1; multiple named layouts allowed. |
| D3 | **Styling = snapshot + manual refresh.** Creating a node copies its NodePrototype's full `style` onto `node.style`. Prototype edits apply only via an explicit **Refresh from prototype**. Render reads `node.style` directly (palette is fallback only); the prototype is **not** consulted at render time. Edges mirror via EdgePrototype → `edge.style`. |
| D4 | **NodePrototype / EdgePrototype** are the two prototype kinds (was `node` / `relationship`). Surfaced as two libraries. |
| D5 | **No entity-level styling. No StyleProfiles.** Remove both entirely. |
| D6 | **Add-node entity picker** — adding a node must choose an existing entity or create a new one (optionally from a NodePrototype), like the existing edge-drop QuickPicker. No anonymous `Node N`. |
| D7 | **Relationships are browsable** — a Relationships list panel mirroring the entity surface. |
| D8 | **Auto-layout uses Dagre** (`@dagrejs/dagre`) from the start. |
| D9 | **Save-as-prototype does not relink** the source entity. |
| D10 | **Palettes demoted** to app-chrome theme + preset/seed source. Concrete snapshots mean a palette swap no longer live-reskins the canvas; per-canvas palette assignment is removed. |
| D11 | **Clean break.** Bump `CURRENT_SCHEMA_VERSION`; **no migration** of existing `.nodge` documents. Add a SQLite migration only so local OPFS dev DBs keep opening. |
| D12 | **Querying is out of scope** (separate follow-up). Keep `src/editor/filter.ts` as the predicate it will build on, but unwire `filter` from rendering. |

Resolution, after refactor:
```
node:  resolveNodeStyle(paletteFallback, node.style)      // node.style seeded from NodePrototype, refreshable
edge:  resolveEdgeStyle(paletteFallback, edge.style)      // edge.style seeded from EdgePrototype, refreshable
```

## 3. Rename map (apply everywhere — schema, gateway, types, UI, tests)

| Current | New |
|---------|-----|
| `Board` / `boardTable` / `board` table | `Diagram` / `diagramTable` / `diagram` |
| `View` / `viewTable` / `view` table | `Layout` / `layoutTable` / `layout` |
| `node.boardId`, `edge.boardId`, `view.boardId` | `…diagramId` |
| `node_position.viewId` | `node_position.layoutId` |
| `entity.prototypeId` | `entity.nodePrototypeId` |
| `relationship.prototypeId` | `relationship.edgePrototypeId` |
| `prototype.kind` value `'relationship'` | `'edge'` (so `PROTOTYPE_KINDS = ['node','edge']`) |
| `node.styleOverride`, `edge.styleOverride` | `node.style`, `edge.style` (now a **full** snapshot, same json type) |
| gateway `*Board*` methods/types | `*Diagram*` |
| gateway `*View*` methods/types | `*Layout*` |

**Deleted:** `styleProfileTable` + `StyleProfile` type + `STYLE_PROFILE_TARGETS`;
`entity.styleOverride`, `entity.styleProfileId`, `node.styleProfileId`, `relationship.styleOverride`,
`prototype.styleProfileId`; gateway `listStyleProfiles`/`createStyleProfile`/`updateStyleProfile`/
`deleteStyleProfile`; `view.paletteId`, `view.filter`; input fields `styleProfileId`,
`styleOverride`, `entityStyleOverride`.

## 4. Model changes (`src/model/`)

Authored once in `schema.ts` (the single source — DDL, JSON validator, and TS types all derive
from it; see the file's header invariant).

- **`diagramTable`** (was `boardTable`): unchanged columns besides the rename.
- **`prototypeTable`**: `kind` enum `['node','edge']`; **remove** `styleProfileId`; `style` is the
  full default style. Public type aliases `NodePrototype` / `EdgePrototype` (both `RowOf<typeof
  prototypeTable>` with a narrowed `kind`, or a discriminated union — implementer's choice).
- **`entityTable`**: `prototypeId`→`nodePrototypeId`; remove `styleOverride`, `styleProfileId`.
- **`relationshipTable`**: `prototypeId`→`edgePrototypeId`; remove `styleOverride`.
- **`nodeTable`**: `boardId`→`diagramId`; `styleOverride`→`style` (full snapshot); remove
  `styleProfileId`.
- **`edgeTable`**: `boardId`→`diagramId`; `styleOverride`→`style`.
- **`layoutTable`** (was `viewTable`): `{ id, diagramId, name, algorithm: enumText(LAYOUT_ALGORITHMS),
  viewport, …timestamps, version }`. `LAYOUT_ALGORITHMS = ['manual','dagre']`. Drop `paletteId`,
  `filter`.
- **`nodePositionTable`**: PK `['layoutId','nodeId']`.
- **Remove** `styleProfileTable`. Update `ALL_TABLES` and the `RowOf` exports.
- **`nested.ts`**: remove `STYLE_PROFILE_TARGETS`, `parseViewFilter` usage from the view (filter
  parser can stay for later querying); add `LAYOUT_ALGORITHMS`.
- **`document.ts`**: `CURRENT_SCHEMA_VERSION` 2→**3**; `boards`→`diagrams`, nested `views`→
  `layouts`; drop top-level `styleProfiles`; rewrite `validateDocument` to the new shape and
  `DocumentDiagram`/`DocumentLayout` interfaces.
- `ddl.ts` regenerates from `ALL_TABLES` — no manual edits.

## 5. SQLite migration (`src/db/migrations.ts`)

Append **v4** (D11). A brand-new DB is created correctly by v1's model-derived `schemaDdl()`; v4
brings *existing* local OPFS DBs forward and is idempotent:
- `CREATE TABLE` the new `layout`, `node_style`-free model (styles live on `node`/`edge`),
  rename/rebuild as needed (SQLite: `ALTER TABLE … RENAME`, `ADD COLUMN`, `DROP COLUMN` ≥3.35).
- `DROP TABLE IF EXISTS style_profile`.
- Drop removed columns (`style_profile_id`, old `style_override` names) where present via the
  existing `addColumnIfMissing`-style guards (add a `dropColumnIfPresent` helper).
- Bump `user_version` to 4; `LATEST_SQLITE_VERSION` follows automatically.

No `.nodge` JSON migration is written (clean break); `io/jsonMigrations.ts` simply targets v3 and
older documents are rejected by `validateDocument`.

## 6. Styling resolution (`src/editor/style.ts`, `diagram.ts`)

- `style.ts`: update the cascade doc to `paletteFallback → node.style` / `→ edge.style`; remove all
  `StyleProfile` references. `resolveNodeStyle`/`resolveEdgeStyle` already accept variadic layers —
  pass only the row's `style`.
- `diagram.ts`: `DiagramSource` drops `styleProfiles`; styles come straight off `node.style` /
  `edge.style`; positions come from the active **layout**. No prototype lookup during render.

## 7. Gateway (`src/gateway/types.ts`, `LocalGateway.ts`) — largest surface

- Apply the rename map to every method, DTO, input/patch, and the `DataGateway` interface.
- **Style writes:** `updateNode({ style })` / `updateEdge({ style })` merge into the row's full
  style (one undoable command each). NodeStylePanel/EdgeStylePanel call these.
- **Snapshot on create:** `addNode` / `placeEntity` / `connectToExistingEntity` /
  `connectToNewEntity` copy the linked **NodePrototype.style** onto `node.style`; edge creation
  (`connectNodes`, `connect*`) copies **EdgePrototype.style** onto `edge.style`. Remove the old
  "seed entity.styleOverride" copy.
- **`refreshFromPrototype` (restored):** re-copy the current prototype `style` onto the target —
  `ids?: Uuid[]` (selected nodes/edges) or `all: true` (every node/edge in a diagram linked to that
  prototype). Returns affected ids. One command.
- **`createPrototypeFromNode` / `createPrototypeFromEdge`:** snapshot `node.style` / `edge.style`
  into a new NodePrototype/EdgePrototype; do **not** relink (D9).
- **Layouts:** `createLayout` / `updateLayout` / `deleteLayout`; **`generateLayout(diagramId,
  layoutId)`** computes positions with **Dagre** and bulk-upserts them (reuse
  `bulkUpsertPositions`). See §8.
- **Remove** all StyleProfile methods; drop removed input fields.
- `getEntityUsages`: `board*`→`diagram*` naming (placements now reference diagrams).

## 8. Auto-layout with Dagre (D8)

- Add dependency **`@dagrejs/dagre`** (maintained fork; `pnpm add @dagrejs/dagre`).
- New pure module `src/editor/autoLayout.ts`: given a diagram's nodes + edges (and optional node
  size hints), build a `dagre.graphlib.Graph`, set `rankdir` (default `'TB'`), run
  `dagre.layout`, and return `{ nodeId, x, y }[]`. Keep it framework-free and unit-tested
  (deterministic positions for a fixed input).
- `generateLayout` in the gateway calls this and persists via `bulkUpsertPositions(layoutId, …)`.
- UI: an **"Auto-arrange"** action on the layout switcher runs `generateLayout` for the active
  layout; React Flow re-renders from the new positions.

## 9. Add-node entity picker (D6)

- Extract a shared **`EntityPicker`** from `QuickPicker.tsx` — the existing/new-entity tabbed
  dialog with search. Props: `entities`, `nodePrototypes`, `onUseExisting(entityId)`,
  `onCreateNew(name, nodePrototypeId | null)`, `onCancel`.
- **Edge-drop** (unchanged behavior): existing→`connectToExistingEntity`, new→`connectToNewEntity`.
- **Add-node** (new): existing→`placeEntity`, new→`addNode(name + nodePrototypeId)`.
- `Editor.tsx`: the Add-mode pane tap (`onPaneClick`), the FAB, and the `addNode` button (≈ lines
  266-289, 365-375, 414-429) open the picker at the target point instead of creating `Node N`.

## 10. Panels / UI (`src/editor/panels/`, `Editor.tsx`)

- **Delete** `StyleProfilePanel.tsx` (+ `.test`); remove its mount and the `style-profiles` query
  in `Editor.tsx`.
- `NodeStylePanel` / `EdgeStylePanel`: write through `updateNode({style})` / `updateEdge({style})`;
  remove the "apply style profile" control; replace the O(graphs×boards) lookup
  (`NodeStylePanel.tsx:67-82`) with the style already on the row; add a **"Refresh from prototype"**
  button calling `refreshFromPrototype`.
- `PrototypePanel`: present **Node prototypes** and **Edge prototypes** as two libraries; keep
  stamp / duplicate / save-as; "Refresh all" → restored `refreshFromPrototype({all})`.
- **New `RelationshipsPanel`**: list the graph's relationships (source→target entity, edge
  prototype, label, metadata) with drill-down to the backing entities/edges.
- `BoardViewBar.tsx`: rename to diagram + **layout switcher** (select/add layout + "Auto-arrange").
- `bootstrap.ts` / `appSettings.ts`: `boardId`/`viewId` → `diagramId`/`layoutId`.

## 12. Verification (end-to-end)

1. `pnpm typecheck && pnpm lint && pnpm test && pnpm build` all green (the pre-push gate).
2. `pnpm dev` manual smoke:
   - Add node → picker → "Create new" w/ a NodePrototype renders in that style; "Use existing"
     places another node of the same entity.
   - Edit a node style → changes; editing the NodePrototype does nothing until **Refresh from
     prototype** re-applies it.
   - Connect two nodes → edge takes its EdgePrototype style; Relationships panel lists it.
   - Add a second Layout → **Auto-arrange** (Dagre) recomputes positions without disturbing the
     manual layout; switching layouts keeps styling identical.
   - Save node as prototype → new NodePrototype captures the look; stamping repeats it.
   - Save → reload: diagram membership, node/edge styles, per-layout positions all persist.

## 13. Suggested sequencing (each a committable, gate-passing chunk)

   bump `CURRENT_SCHEMA_VERSION`; update `document.ts`. Fix `model.test`.
2. **SQLite migration v4** + `db` tests.
3. **Gateway** — rename map, style writes, snapshot-on-create, restored `refreshFromPrototype`,
   layout CRUD; update gateway tests. (Largest chunk — consider splitting rename vs behavior.)
4. **Dagre auto-layout** — dependency, `autoLayout.ts`, `generateLayout`, tests.
5. **Style/diagram resolution** — `style.ts` + `diagram.ts`; `style.test`/`diagram.test`.
6. **Add-node EntityPicker** — extract + wire both flows; `Editor`/picker tests.
7. **Panels** — delete StyleProfilePanel; update style panels (+refresh button); split prototype
   libraries; new RelationshipsPanel; layout switcher; component tests.
8. **Full sweep** — run the gate, manual smoke (§12), update this checklist.
